[[kwctl-bench]]
=== `kwctl bench`

EvalĂşa una directiva {short-project-name}.

La directiva se puede especificar de las siguientes maneras:
- URI: p. ej., `registry://ghcr.io/kubewarden/policies/psp-policy:latest` o `\https://example.com/kubewarden/policies/main/psp-policy/psp-policy.wasm`
- Prefijo SHA: p. ej., `c3b80a10f9c3` (requiere que la directiva ya estĂ© descargada)
- Archivo WASM local: p. ej., `file://home/tux/new-policy/psp-policy.wasm`
- Archivo YAML local: p. ej., `file://home/tux/cluster-admission-policy.yaml` (contiene declaraciones de Recursos Personalizados {short-project-name} como `ClusterAdmissionPolicy`, `AdmissionPolicy`, etc.)

Comportamiento por defecto:
Si se omite el esquema, se asume `file://`, enraizado en el directorio actual.

Notas sobre Recursos Personalizados {short-project-name}:
- Las banderas `--request-path`, `--settings-path` y `--settings-json` son ignoradas; los ajustes se leen de la definiciĂłn del Recurso Personalizado.
- La bandera `--execution-mode` se aplica a todas las directivas en el archivo YAML.
- La bandera `--raw` no se puede usar, ya que los Recursos Personalizados de {short-project-name} no soportan directivas `raw`.

Solo se evalĂşan los siguientes atributos de la DefiniciĂłn de Recurso Personalizado (CRD):
- MĂłdulo de directiva
- Ajustes de directiva
- Recursos contextuales a los que la directiva puede acceder

Otros campos, como `rules`, `matchConditions`, `objectSelector` y `namespaceSelector`, son ignorados.

Un archivo YAML puede contener mĂşltiples declaraciones de Recursos Personalizados. En este caso, `kwctl` evalĂşa cada directiva en el archivo utilizando la misma solicitud durante cada evaluaciĂłn.

*Uso:* `kwctl bench [OPTIONS] --request-path <PATH> <uri_or_sha_prefix_or_yaml_file>`

==== Argumentos

* `<URI_OR_SHA_PREFIX_OR_YAML_FILE>` â€” URI de la directiva, prefijo SHA o archivo YAML que contiene recursos de directiva {short-project-name}. Esquemas soportados: registry://, https://, file://. Si se omite el esquema, se asume file://, enraizado en el directorio actual.

==== Opciones

* `--allow-context-aware <ALLOW-CONTEXT-AWARE>` â€” Concede acceso a los recursos de Kubernetes definidos dentro de la secciĂłn `contextAwareResources` de la directiva. Advertencia: revisa la lista de recursos cuidadosamente para evitar abusos. Deshabilitada por defecto
* `--cert-email <VALUE>` â€” Correo electrĂłnico esperado en el certificado Fulcio
* `--cert-oidc-issuer <VALUE>` â€” Emisor OIDC esperado en los certificados Fulcio
* `--disable-wasmtime-cache <DISABLE-WASMTIME-CACHE>` â€” Desactiva el uso de la cachĂ© de wasmtime
* `--docker-config-json-path <PATH>` â€” Ruta a un directorio que contiene el archivo 'config.json' de Docker. Se puede usar para indicar detalles de autenticaciĂłn del registro
* `--dump-results-to-disk <DUMP_RESULTS_TO_DISK>` â€” Coloca los resultados en target/tiny-bench/label/.. si se puede encontrar target. utilizado para comparar ejecuciones anteriores
* `-e`, `--execution-mode <MODE>` â€” El tiempo de ejecuciĂłn a utilizar para ejecutar esta directiva +
  Valores posibles: `opa`, `gatekeeper`, `kubewarden`, `wasi`
* `--github-owner <VALUE>` â€” Propietario de GitHub esperado en los certificados generados en las canalizaciones de CD
* `--github-repo <VALUE>` â€” Repositorio de GitHub esperado en los certificados generados en las canalizaciones de CD
* `--measurement-time <SECONDS>` â€” CuĂˇnto tiempo 'deberĂ­a' ejecutarse el banco, num_samples tiene prioridad, por lo que la evaluaciĂłn tomarĂˇ mĂˇs tiempo para poder recoger num_samples si el cĂłdigo a evaluar es mĂˇs lento que este lĂ­mite de tiempo permitido
* `--num-resamples <NUM>` â€” CuĂˇntas re-muestras deberĂ­an hacerse
* `--num-samples <NUM>` â€” CuĂˇntas re-muestras deberĂ­an hacerse. Se recomienda al menos 50, mĂˇs de 100 no parece dar un resultado significativamente diferente
* `--raw <RAW>` â€” Valida una solicitud en bruto +
  Valor por defecto: `false`
* `--record-host-capabilities-interactions <FILE>` â€” Registra todas las comunicaciones de capacidades de directivas y hosts en el archivo dado. Ăštil para combinar mĂˇs tarde con la bandera '--replay-host-capabilities-interactions'
* `--replay-host-capabilities-interactions <FILE>` â€” Durante los intercambios de directivas y capacidades de hosts, el host devuelve las respuestas encontradas dentro del archivo proporcionado. Esto es Ăştil para probar directivas de manera reproducible, dado que no se realizan interacciones externas con registros OCI, DNS, Kubernetes.
* `-r`, `--request-path <PATH>` â€” Archivo que contiene el objeto de solicitud de admisiĂłn de Kubernetes en formato JSON
* `--settings-json <VALUE>` â€” Cadena JSON que contiene la configuraciĂłn para esta directiva
* `-s`, `--settings-path <PATH>` â€” Archivo que contiene la configuraciĂłn para esta directiva
* `--sigstore-trust-config <PATH>` â€” Archivo en formato JSON que se ajusta al mensaje ClientTrustConfig en las especificaciones protobuf de Sigstore. Este archivo configura todo el estado de la instancia de Sigstore, incluyendo las URIs utilizadas para acceder a los servicios de CA y transparencia de artefactos, asĂ­ como la raĂ­z de confianza criptogrĂˇfica en sĂ­.
* `--sources-path <PATH>` â€” Archivo YAML que contiene informaciĂłn de origen (https, hosts inseguros de registro, CA personalizadas...)
* `-a`, `--verification-annotation <KEY=VALUE>` â€” AnotaciĂłn en formato clave=valor. Puede repetirse mĂşltiples veces
* `--verification-config-path <PATH>` â€” Archivo YAML que contiene informaciĂłn de configuraciĂłn de verificaciĂłn (firmas, claves pĂşblicas...)
* `-k`, `--verification-key <PATH>` â€” Ruta a la clave utilizada para verificar la directiva. Puede repetirse mĂşltiples veces
* `--warm-up-time <SECONDS>` â€” CuĂˇnto tiempo debe calentarse el benchmark
